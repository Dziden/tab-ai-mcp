# Архитектура ТАБ:Библиотека ИИ (tab_mcp)

## Компоненты

| Компонент | Описание | Размещение |
|---|---|---|
| **БИИ** | ТАБ:Библиотека ИИ — расширение 1С, предоставляет HTTP-сервис | CPU, внутри 1С |
| **База 1С** | 1С:Предприятие — источник данных, OData REST API | — |
| **tab_mcp** | MCP-коннектор к 1С. Три инструмента: read_1c, write_1c, count_document_marks | CPU |
| **tab_ss** | Хранит credentials 1С. Семантический поиск, прогнозирование, аномалии, RAG. Встроенный LLM при наличии GPU | CPU / GPU |
| **tab_ca** | Мультиагент LLM | GPU |
| **Элемент** | Биллинг и тарификация (cloud-режим) | CPU |
| **LLM провайдеры** | GigaChat, OpenAI, DeepSeek и др. | Cloud / GPU |

---

## Поток credentials 1С

Credentials (URL базы, логин, пароль) хранятся в tab_ss и получаются per-request:

```
Администратор → POST /v1/onec/connections → tab_ss (хранит зашифрованно)

ЛЛМ вызывает read_1c(organization, query, ...)
        │
        ▼
    tab_mcp → POST /v1/onec/connections/resolve?organization=X&user_id=Y → tab_ss
                    │
                    ▼ (odata_base_url, login, password)
    tab_mcp → OData → База 1С
```

Credentials кешируются в памяти 5 минут. Изоляция по `(organization, user_id)`.

## Два пути взаимодействия с 1С

### Путь 1 — Чтение через OData

```
tab_ss → tab_mcp → OData → База 1С
```

- OData — стандартный REST API 1С, не требует БИИ
- Используется для автоматического получения данных (индексация в векторную БД, RAG, fine-tuning)
- Если OData недоступен — прикладной разработчик сам делает выборку и отправляет данные в tab_ss

### Путь 2 — Запись через OData

```
tab_ss → tab_mcp → OData (PATCH / POST) → База 1С
```

- write_1c напрямую создаёт или обновляет объекты через OData
- Если есть `Ref_Key` в данных → PATCH (обновление), иначе → POST (создание)

---

## Режимы работы БИИ

| Функция | Данные 1С нужны | Путь |
|---|---|---|
| Переводчик текста | Нет | БИИ → tab_ss напрямую |
| OCR | Нет | БИИ → tab_ss напрямую |
| Проверка подписи документа | Нет | БИИ → tab_ss напрямую |
| Семантический поиск | Да (через OData) | БИИ → tab_ss → tab_mcp → OData |
| Поиск аномалий | Да (разработчик сам отбирает) | БИИ → tab_ss (данные в теле запроса) |
| Прогноз динамики | Да (разработчик сам отбирает) | БИИ → tab_ss (данные в теле запроса) |

## Инструменты tab_mcp

### `read_1c`

Читает данные из 1С через OData.

```
Вход:  organization, query, filter?, select?, expand?, top?, skip?
Выход: список объектов
```

`query` — точное OData-имя, виртуальная таблица или описание на русском языке:
- `"Catalog_Номенклатура"` — точное имя
- `"AccountingRegister_Хозрасчетный/Balance(Period=datetime'2025-12-31T00:00:00')"` — виртуальная таблица
- `"остатки товаров на складах"` — резолвится через семантический поиск

### `write_1c`

Записывает данные в 1С через OData (upsert).

```
Вход:  organization, query, data (объект или список)
Выход: {"written": N, "items": [...]}
```

- `Ref_Key` в data → PATCH (обновление существующего)
- Нет `Ref_Key` → POST (создание нового)

### `count_document_marks`

Подсчитывает печати и подписи на скане или ЭДО-документе.

```
Вход:  document_base64 (base64 JPG/PNG или PDF)
Выход: {document_type, contractors: [{name, inn, stamps, signatures}], ...}
```

## Схема (текстовая)

```
                    [LLM провайдеры]
                     ↑           ↑
                  cloud        cloud
                     |           |
[tab_ca GPU] ←─ onprem ─→ [tab_ss CPU] ←─── [tab_mcp CPU]
                                │                   │
                             credentials          OData
                             (resolve)              │
                                └──────────────► [База 1С]
```

## Биллинг (Cloud)

В облачном режиме все запросы БИИ → tab_ss проходят через Элемент, который фиксирует каждый вызов и списывает стоимость согласно тарифу.
